2Ô∏è‚É£ CI/CD: The Factory of Your Product

The Factory Is The Product

As Elon Musk said: "The factory is the product."

In software, the factory is your CI/CD pipeline.

Bad pipeline:

  • Slow
  • Manual
  • Unreliable
  • Difficult to rollback

Result: Product is slow, fragile, hard to improve.

Good pipeline:

  • Automated
  • Fast
  • Reproducible
  • Easy rollback

Result: Product improves automatically.

CI ‚Äî Continuous Integration

Every change you make:

  1. Automatically tested
  2. Automatically built
  3. Automatically validated

Goal: Detect errors early.

CD ‚Äî Continuous Delivery / Deployment

After CI passes:

  1. Automatically deployed to staging, OR
  2. Ready to deploy to production with one click

Goal: Deliver value continuously.

3Ô∏è‚É£ Step 1: Reproducible Build

Core principle: If you can't build it from scratch, you can't deploy it.

Checklist:

  • ‚úÖ Use npm ci instead of npm install
  • ‚úÖ Pin all dependency versions
  • ‚úÖ No hidden local dependencies
  • ‚úÖ Build produces clear artifacts (dist/, Docker image, etc.)
  • ‚úÖ Same build works on local, CI, and production

Senior rule:

If it doesn't work on another machine the same way, it's not ready.

4Ô∏è‚É£ Step 2: Separate Configuration from Code

Never hardcode:

  • URLs
  • Secrets
  • Passwords
  • API keys

Use environment variables:

API_URL=https://api.example.com
DATABASE_URL=postgresql://user:pass@host/db
JWT_SECRET=your-secret-key-here
LOG_LEVEL=info

Each environment has different values:

  • local
  • staging
  • production

This follows the 12-Factor App principle.

5Ô∏è‚É£ Step 3: Containerization (Docker)

Why Docker?

Because:

  • Guarantees same environment in local, CI, and production
  • Eliminates "works on my machine"
  • Converts your app into a portable unit
  • Makes deployment deterministic

Basic Dockerfile

FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .

RUN npm run build

EXPOSE 3000

CMD ["npm", "start"]

Key principles:

  • Start with specific base image version (not `latest`)
  • Use npm ci not npm install
  • Run build in container
  • Container size matters (use Alpine Linux)

Minimum professional CI pipeline:

Pull Request ‚Üí
  Lint (Code Quality) ‚Üí

Example GitHub Actions

name: CI

on:
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

Best Practices

  • ‚úÖ Fail if tests fail
  • ‚úÖ Don't merge if CI fails
  • ‚úÖ Cache dependencies
  • ‚úÖ Run in parallel when possible
  • ‚úÖ Keep total time under 10 minutes

8Ô∏è‚É£ Step 6: Observability (Logging, Metrics, Alerts)

This is where many developers fail.

In production, you don't debug.

You observe.

üîç 1. Structured Logs

{
  "level": "error",
  "timestamp": "2025-02-18T10:30:45.123Z",
  "requestId": "req-123-456",
  "endpoint": "/api/users",
  "method": "POST",
  "statusCode": 500,
  "message": "Database connection timeout",
  "stack": "..."
}

Never log like this:

console.log("Error:" + error)

Always use structured logging:

logger.error({
  message: "Database connection timeout",
  error: error,
  requestId: req.id,
  endpoint: req.path
})

üìä 2. Metrics

Track what matters:

  • Response time (latency)
  • Error rate
  • Requests per second
  • CPU usage
  • Memory usage
  • Database connections

üö® 3. Alerts

Automatically notify when things go wrong:

  • Error rate > 5%
  • Response time > 2s (p95)
  • CPU > 80%
  • Service is down
  • Disk space < 10%

Senior rule:

If you can't measure it, you can't operate it.

üîü Step 8: Deployment Strategies

üîµ Blue/Green Deployment

Two identical environments.

  • Blue = Current production
  • Green = New version

Test in Green, then switch traffic.

If problems, instant rollback to Blue.

üü° Canary Release

Send 5% of traffic to new version.

Monitor metrics.

If healthy, gradually increase to 100%.

If problems, instant rollback.

üü¢ Rolling Deployment

Update 1 instance at a time.

Keep service up the whole time.

Slower but no downtime.

1Ô∏è‚É£1Ô∏è‚É£ Step 9: Rollback Strategy

Operational excellence = Recovery speed.

Minimum capability:

Re-deploy previous version in < 5 minutes.

How:

  • Keep Docker images tagged by version
  • Have previous deployment config stored
  • One-click rollback button
  • Automated health checks post-rollback

1Ô∏è‚É£2Ô∏è‚É£ Step 10: Security Basics

You don't need to be a security expert, but you must:

  • ‚úÖ Use HTTPS everywhere
  • ‚úÖ Manage secrets correctly (use vaults, never in code)
  • ‚úÖ Don't expose databases directly
  • ‚úÖ Apply principle of least privilege

üîê Principle of Least Privilege

Your application should only have access to:

Exactly what it needs.

Nothing more.

Bad:

App has root access to entire infrastructure.

Good:

App can only:

  • Read from specific database table
  • Write to specific S3 bucket
  • Read from specific secrets vault
  • Nothing else

If compromised, damage is limited.

1Ô∏è‚É£6Ô∏è‚É£ Interview Questions

‚ùì What is CI/CD?

CI is automatically testing and building every change.
CD is automatically deploying after validation.

‚ùì What's the difference between Blue/Green and Canary?

Blue/Green switches all traffic at once with instant rollback.
Canary gradually increases traffic, monitoring metrics.

‚ùì Why should you not rebuild for production?

Because different builds might behave differently due to environment differences.
Always promote the same tested artifact.

‚ùì What's the principle of least privilege?

Give applications only the permissions they absolutely need.
Limits damage if compromised.

‚ùì Why are health checks important?

Enable automatic restarts, load balancers know when to send traffic,
orchestrators can self-heal.
