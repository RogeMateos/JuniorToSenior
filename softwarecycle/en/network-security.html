When you move from developing code to deploying in the cloud, something important happens:

You're now responsible for making sure no one accesses things they shouldn't.

Locally everything works because:

  • Everything is on localhost
  • There's no public network
  • There are no attackers

In production:

  • There's the internet
  • Bots are scanning ports
  • There are brute force attempts
  • There's risk of data leakage

That's why VPCs and network mechanisms exist.

Within the VPC you create subnets.

  • It has a route table with:
    • 0.0.0.0/0 ‚Üí Internet Gateway

It's "private" when:

  • It doesn't have a direct route to the Internet Gateway

5Ô∏è‚É£ What Makes Something Really Public?

Many juniors believe:

"It's on AWS, so it's already public"

No.

For a resource to be accessible from the internet it needs:

  • Route to Internet Gateway
  • Public IP
  • Security Group that allows inbound
  • Service listening on that port

If one of those fails ‚Üí it's not accessible.

This is super important for debugging.

üîπ Inbound vs Outbound

Inbound = what can enter
Outbound = what can leave

Typical backend example:

Inbound:

  • 443 from Load Balancer
  • 22 only from your IP (if using SSH)

Outbound:

  • 5432 to database
  • 443 to internet (npm install, external APIs)

7Ô∏è‚É£ Security Groups Are STATEFUL

This is key and many people ignore it.

If you allow inbound:

The return traffic is automatically allowed.

You don't need to open the return port.

9Ô∏è‚É£ Most Important Principle: Least Privilege

Never open more than necessary.

Incorrect example (very common in juniors):

Port 5432
Source: 0.0.0.0/0

That leaves your database open to the internet.

Correct:

Port 5432
Source: Backend-SG

That means:

Only servers that have that SG can connect.

This is called Security Group referencing.

It's real professional security.

1Ô∏è‚É£1Ô∏è‚É£ How Does Backend Get to Internet If It's Private?

Enter:

NAT Gateway

The backend in private subnet:

  • Can't receive traffic from the internet
  • But can exit to the internet through NAT

Why does this matter?

Because:

  • You need npm install
  • You need to call external APIs
  • You need to download updates

But you don't want anyone from outside to initiate a connection to you.

1Ô∏è‚É£3Ô∏è‚É£ How to Debug a Timeout

This saves you hours.

If you can't connect to the DB:

First ask yourself:

  • Is it a password error?
    • ‚Üí You reached the DB
  • Is it a timeout?
    • ‚Üí You're not passing the firewall

Checklist:

  • Does DB SG allow 5432 from backend SG?
  • Does backend really use that SG?
  • Are they in the same VPC?
  • Is there a correct route?
  • Is DB private (correct)?
  • Are you trying to access from outside?

Practical rule:

Timeout = Networking
Auth error = Credentials

It's restricted to:

Because you can also have:

‚Üí data exfiltration

Security is not just inbound.
